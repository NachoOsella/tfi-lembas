# Validaciones criticas

## Backend obligatorias

| Area | Validacion |
|---|---|
| Usuarios | Usuario activo, rol valido y permisos suficientes. |
| Productos | Codigo de barras unico en activos, precio no negativo, nombre obligatorio. |
| Stock | Cantidad positiva, stock suficiente, lote requerido si maneja vencimiento. |
| FEFO | Ordenar lotes por vencimiento y evitar lote negativo. |
| Venta | Estado BORRADOR para modificar/confirmar, total consistente, stock disponible. |
| Precio en venta | Motivo obligatorio si cambia precio, permiso requerido. |
| Ofertas | Fechas validas, oferta activa y no vencida. |
| Caja | Monto positivo, concepto obligatorio en manuales, turno abierto si aplica. |
| Consumo | Razon obligatoria, OTRO con descripcion, no caja. |
| Compras | Proveedor obligatorio, cantidad esperada/recibida positiva, vencimiento opcional con advertencia si el producto lo controla, suma de lotes igual a recibido, solo RECIBIDA ingresa stock. |
| Listas | Proveedor obligatorio, fuente permitida, items revisados antes de aplicar. |
| Pricing | Aprobacion explicita para cambios de precio. |
| Etiquetas | Marcar pendiente cuando cambia precio u oferta. |

## Validaciones de transaccion

- Si falla descuento de stock, no se confirma venta.
- Si falla caja, no se confirma venta.
- Si falla actualizacion de precio en lista, no se marca lista como aplicada.
- Si un consumo requiere aprobacion, no descuenta stock hasta aprobarse.
- Si una compra queda en borrador, no genera ingreso de stock.
- Si falla el ingreso de stock de una compra, no se marca como recibida.
- La cantidad ingresada a stock en una compra debe ser la cantidad recibida real.
- Si una linea se divide en lotes, la suma de lotes debe coincidir con la cantidad recibida.

## Validaciones de interfaz

- Mensajes claros para producto no encontrado por scanner.
- Advertencia antes de anular venta.
- Confirmacion antes de aplicar cambios masivos de precios.
- Mostrar diferencia de caja antes de cerrar.
