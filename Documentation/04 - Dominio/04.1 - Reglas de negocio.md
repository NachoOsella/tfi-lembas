# Reglas de negocio

## Usuarios y permisos

| ID | Regla |
|---|---|
| RN-01 | Todo usuario debe autenticarse para operar. |
| RN-02 | Un usuario inactivo no puede iniciar sesion ni registrar operaciones. |
| RN-03 | La administradora puede acceder a configuracion, reportes, auditoria, caja completa y gestion general. |
| RN-04 | El empleado solo accede a funciones operativas habilitadas. |
| RN-05 | Las acciones sensibles registran usuario, fecha, entidad y detalle. |
| RN-06 | Costos, margenes, reportes financieros y auditoria completa no se muestran al empleado salvo permiso. |

## Productos

| ID | Regla |
|---|---|
| RN-07 | Un producto con historial no se elimina fisicamente. |
| RN-08 | La baja de producto es logica. |
| RN-09 | Un producto inactivo no puede venderse ni usarse en nuevas operaciones normales. |
| RN-10 | El codigo de barras es opcional, pero no puede repetirse entre productos activos. |
| RN-11 | El precio de venta no puede ser negativo. |
| RN-12 | Todo cambio de precio de venta guarda historial. |
| RN-13 | El costo estimado es visible solo para roles autorizados. |
| RN-14 | Stock minimo e ideal deben ser mayores o iguales a cero. |

## Stock y FEFO

| ID | Regla |
|---|---|
| RN-15 | El stock disponible se calcula desde lotes y movimientos. |
| RN-16 | Todo ingreso de stock identifica producto, cantidad, usuario, fecha y origen. |
| RN-17 | Si el producto controla vencimiento, el ingreso debe permitir registrar fecha/lote y advertir si se confirma sin fecha. |
| RN-18 | Ninguna operacion normal debe dejar cantidad de lote negativa. |
| RN-19 | No se permite vender, consumir o retirar mas stock que el disponible salvo autorizacion auditada. |
| RN-20 | Las salidas aplican FEFO priorizando lotes con fecha mas proxima; los lotes sin vencimiento quedan al final. |
| RN-21 | Todo ajuste manual requiere motivo. |
| RN-22 | Venta, consumo interno, merma, vencimiento y ajuste son causas distintas. |
| RN-23 | Stock bajo se determina con stock minimo. |
| RN-24 | Vencimiento proximo se determina con umbral configurable. |
| RN-25 | Los movimientos de stock no se editan libremente; las correcciones son compensatorias. |

## Ventas

| ID | Regla |
|---|---|
| RN-32 | Una venta inicia en borrador y solo impacta al confirmarse. |
| RN-33 | Una venta confirmada genera egreso de stock. |
| RN-34 | Una venta confirmada genera ingreso de caja. |
| RN-35 | Confirmar venta es transaccional: stock, caja, estado y auditoria se confirman o revierten juntos. |
| RN-36 | La venta no se elimina fisicamente; se anula o compensa. |
| RN-37 | Toda anulacion conserva motivo, usuario y fecha. |
| RN-38 | El scanner se trata como entrada de teclado. |
| RN-39 | Productos sin codigo se buscan manualmente. |
| RN-40 | La cantidad en venta debe ser mayor que cero. |
| RN-41 | Agregar un item al borrador no descuenta stock. |

## Precios y ofertas

| ID | Regla |
|---|---|
| RN-42 | Cada detalle guarda precio base historico. |
| RN-43 | Cada detalle guarda precio aplicado final. |
| RN-44 | Si el precio aplicado difiere del base, se exige motivo. |
| RN-45 | Modificar precio depende de permisos o limites configurados. |
| RN-46 | Toda modificacion manual de precio se audita. |
| RN-47 | Cambios posteriores de precio no alteran ventas historicas. |
| RN-50 | Una oferta tiene producto, fecha de inicio y fecha de fin. |
| RN-51 | La fecha de fin no puede ser anterior al inicio. |
| RN-52 | Una oferta inactiva o vencida no se aplica automaticamente. |
| RN-53 | Si hay multiples ofertas, se usa prioridad definida. |
| RN-54 | La venta guarda oferta aplicada y precio final. |

## Consumo interno, mermas y vencimientos

| ID | Regla |
|---|---|
| RN-58 | El consumo interno descuenta stock. |
| RN-59 | El consumo interno no genera caja. |
| RN-60 | Todo consumo registra usuario, producto, cantidad, razon y fecha. |
| RN-61 | La razon OTRO exige descripcion. |
| RN-62 | Un consumo rechazado no descuenta stock. |
| RN-66 | Una merma descuenta stock y exige motivo. |
| RN-67 | Un retiro por vencimiento identifica lote o fecha afectada. |
| RN-68 | Mermas y vencimientos no generan caja. |

## Caja

| ID | Regla |
|---|---|
| RN-70 | La caja/turno debe estar abierto si la configuracion lo exige. |
| RN-71 | Todo ingreso automatico por venta referencia la venta confirmada. |
| RN-72 | Ingresos y egresos manuales requieren concepto y responsable. |
| RN-73 | Los egresos manuales requieren monto positivo. |
| RN-74 | El cierre compara saldo esperado y saldo contado. |
| RN-75 | Toda diferencia queda registrada. |
| RN-76 | Un cierre confirmado solo se corrige con permiso administrativo y auditoria. |

## Proveedores, listas y pricing

| ID | Regla |
|---|---|
| RN-79 | Un proveedor inactivo conserva historial. |
| RN-80 | Toda lista se asocia a proveedor, fecha y usuario de carga. |
| RN-81 | Las fuentes permitidas son Excel, texto WhatsApp y carga manual. |
| RN-82 | PDF, imagenes, fotos, OCR, audio y catalogos web quedan fuera del MVP. |
| RN-83 | Una lista no modifica costos ni precios sin revision. |
| RN-84 | Cada item tiene estado de reconocimiento. |
| RN-85 | Items dudosos se resuelven manualmente o se ignoran. |
| RN-86 | Costo proveedor y precio de venta son conceptos distintos. |
| RN-87 | Todo costo aplicado guarda historial por proveedor. |
| RN-88 | Todo precio de venta aplicado guarda historial. |
| RN-89 | Los cambios masivos requieren confirmacion explicita. |
| RN-90 | Comparar proveedores debe mostrar fecha de actualizacion del costo. |

## Compras simples

| ID | Regla |
|---|---|
| RN-91 | Una compra simple debe estar asociada a un proveedor. |
| RN-92 | Una compra puede guardarse como borrador antes de confirmar recepcion. |
| RN-93 | Una compra en borrador representa lo esperado y no impacta stock. |
| RN-94 | Cada producto precargado debe tener cantidad esperada mayor que cero. |
| RN-95 | Al recibir, la cantidad que ingresa a stock es la cantidad recibida real, no la esperada. |
| RN-96 | La cantidad recibida puede ser menor, igual o mayor que la esperada, dejando registrada la diferencia. |
| RN-97 | La fecha de vencimiento y el lote se cargan manualmente al recibir y son opcionales por defecto. |
| RN-98 | Si un producto controla vencimiento y se confirma sin fecha, el sistema debe advertir que no generara alertas de vencimiento para ese lote. |
| RN-99 | Si una linea se divide en varios lotes, la suma de cantidades de lotes debe coincidir con la cantidad recibida. |
| RN-100 | Al confirmar una compra se generan lotes y movimientos de stock `INGRESO_COMPRA`. |
| RN-101 | Los lotes sin fecha de vencimiento no generan alertas de vencimiento y quedan al final del criterio FEFO. |
| RN-102 | La compra simple no reemplaza factura fiscal ni registra impuestos, pagos, deuda o cuenta corriente. |
| RN-103 | Una compra recibida no se edita libremente; las correcciones se realizan con ajuste de stock o anulacion controlada. |
| RN-103.1 | El costo unitario recibido puede actualizar el ultimo costo de ProductoProveedor y debe auditarse. |

## Reportes, alertas y auditoria

| ID | Regla |
|---|---|
| RN-104 | Reportes derivan de datos persistidos en backend. |
| RN-105 | Reportes respetan permisos de usuario. |
| RN-106 | Alertas usan umbrales configurables. |
| RN-107 | Alertas no deben duplicarse si la condicion no cambio. |
| RN-111 | La auditoria registra acciones criticas. |
| RN-112 | La auditoria no es editable por usuarios comunes. |
| RN-113 | Deben auditarse ventas, anulaciones, stock, precios, ofertas, consumos, caja e importaciones. |
