# Requisitos no funcionales

## Usabilidad

| ID | Requisito |
|---|---|
| RNF-01 | La interfaz debe ser responsive y priorizar uso mobile-first. |
| RNF-02 | La venta mostrador debe permitir operacion rapida con scanner y teclado. |
| RNF-03 | Las acciones criticas deben mostrar confirmacion clara antes de persistir. |
| RNF-04 | Los errores deben indicar causa y posible correccion sin exponer detalles tecnicos. |
| RNF-05 | Las vistas de empleado deben ocultar informacion sensible no necesaria. |

## Seguridad

| ID | Requisito |
|---|---|
| RNF-06 | Las contraseñas deben almacenarse con hash seguro. |
| RNF-07 | La autorizacion debe validarse en backend, no solo en frontend. |
| RNF-08 | Los endpoints administrativos deben requerir rol o permiso especifico. |
| RNF-09 | El sistema debe aplicar CORS restringido por ambiente. |
| RNF-10 | La auditoria no debe poder editarse por usuarios comunes. |

## Consistencia e integridad

| ID | Requisito |
|---|---|
| RNF-11 | Las operaciones venta-stock-caja deben ser transaccionales. |
| RNF-12 | La aplicacion de listas de precios debe ser transaccional. |
| RNF-13 | Debe existir integridad referencial en la base de datos. |
| RNF-14 | Los estados de entidades transaccionales deben estar controlados. |
| RNF-15 | Las eliminaciones de entidades con historial deben ser logicas. |

## Rendimiento y disponibilidad

| ID | Requisito |
|---|---|
| RNF-16 | Las busquedas por codigo de barras deben responder rapidamente en condiciones normales de local. |
| RNF-17 | Las consultas frecuentes deben contar con indices adecuados. |
| RNF-18 | La aplicacion debe poder ejecutarse en entorno local de desarrollo con Docker Compose. |
| RNF-19 | El despliegue debe permitir acceder desde dispositivos del local mediante navegador. |

## Mantenibilidad

| ID | Requisito |
|---|---|
| RNF-20 | El backend debe organizarse por modulos de dominio. |
| RNF-21 | Los controllers deben delegar reglas de negocio en servicios. |
| RNF-22 | La API debe usar DTOs y no exponer entidades directamente. |
| RNF-23 | El esquema de base de datos debe versionarse con migraciones. |
| RNF-24 | Las reglas criticas deben tener pruebas automatizadas. |
| RNF-25 | La documentacion debe mantenerse modular y trazable. |

## Portabilidad y despliegue

| ID | Requisito |
|---|---|
| RNF-26 | El sistema debe poder construirse con comandos documentados. |
| RNF-27 | La configuracion sensible debe venir de variables de entorno. |
| RNF-28 | Nginx debe actuar como reverse proxy en despliegue productivo o demostrativo. |
| RNF-29 | La documentacion entregable no debe incluir carpetas locales `.obsidian` ni `.pi`. |

## Observabilidad y auditoria

| ID | Requisito |
|---|---|
| RNF-30 | Los errores de negocio deben registrarse de forma controlada. |
| RNF-31 | Las acciones criticas deben generar auditoria con usuario, entidad, fecha y detalle. |
| RNF-32 | Los reportes deben derivarse de datos persistidos en backend. |
